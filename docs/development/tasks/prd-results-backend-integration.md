# PRD: Results Page — Backend Integration

**Version:** 1.0
**Status:** Ready for Implementation
**Scope:** Backend + Frontend

---

## 1. Introduction / Overview

The Results page currently displays hardcoded mock data. This feature replaces all mock values with real data sourced from the backend: actual player submissions (images), real voter star averages from the blind voting phase, and stub AI judge scores (randomized on the server, with a clear hook for future Claude API integration).

The results must also be persistable — players who refresh the page or revisit the challenge code later must be able to retrieve the final standings without re-playing the game.

---

## 2. Goals

1. **Real scores** — The leaderboard ranks players by `(voterAvg + aiScore) / 2` computed from actual votes cast during the blind voting phase.
2. **Real images** — Each player's submission thumbnail and expanded view load from the backend's static file server (`/uploads/{filename}`).
3. **Stub AI scores** — A deterministic mock AI score is generated server-side per submission (random float in range `3.0–5.0`, seeded per submission ID for consistency). The scoring function is isolated in its own module so it can be swapped for a real Claude API call later without touching the rest of the flow.
4. **Persistable results** — A new REST endpoint (`GET /api/challenges/:code/results`) returns the final standings so refreshing or revisiting the page still shows the correct data.
5. **Zero mock data in ResultsPage** — All `MOCK_RESULTS` constants are removed from the frontend.

---

## 3. User Stories

- **As a player**, after the last design is voted on, I want to see the real final rankings — including my own score — so I know who won.
- **As a player**, I want to see thumbnail previews of the top 3 designs (my opponents' actual submitted work) and be able to expand them.
- **As a player**, if I accidentally refresh the results page, I want the standings to still be there — not a blank page.
- **As a developer**, I want the AI scoring function isolated in one file so it's easy to replace with a real Claude API call in the future.

## 4. Functional Requirements

### 4.1 Backend — AI Score Generation

1. The system **must** create a new module at `backend/src/services/aiJudge.js` that exports a single function: `scoreSubmission(submissionId, spec)`.
2. The initial implementation **must** return a deterministic mock score: a float between `3.0` and `5.0`, rounded to one decimal place, derived from a simple hash of `submissionId` so the same submission always returns the same score.
3. The function signature **must** accept `(submissionId, spec)` so that a future implementation can pass both the image identifier and the design brief to a Claude vision API call.
4. The function **must** include a prominent comment block: `// TODO: Replace mock with real Claude API call — see docs/development/prd.md §3.5`.

### 4.2 Backend — Score Computation Update

5. The `Room.computeScores()` method **must** be updated to include an `aiScore` field per player, generated by calling `scoreSubmission(player.submissionId, this.spec)`.
6. The final `combinedScore` for ranking **must** be calculated as: `(voterAvg + aiScore) / 2`, rounded to two decimal places.
7. The scores array returned by `computeScores()` **must** have the following shape per entry:
   ```json
   {
     "playerId": "socket-id",
     "nickname": "Alex Chen",
     "submissionId": "uuid.png",
     "imageUrl": "/uploads/uuid.png",
     "voterScore": 4.6,
     "aiScore": 4.3,
     "combinedScore": 4.45,
     "rank": 1
   }
   ```
8. Entries **must** be sorted by `combinedScore` descending before `rank` is assigned.
9. Players with no submission (`submissionId === null`) **must** receive `voterScore: 0`, `aiScore: 0`, `combinedScore: 0`, and be ranked last.

### 4.3 Backend — Results Persistence

10. The system **must** store the computed scores on the `Room` instance after `computeScores()` is called: `room.finalScores = scores`.
11. The system **must** add a new REST route: `GET /api/challenges/:code/results`.
12. The route **must** return `200` with the stored `room.finalScores` array if the room exists and `room.phase === 'results'`.
13. The route **must** return `404 { error: 'Results not available' }` if the room does not exist or the game has not yet reached the results phase.
14. The `imageUrl` in every scores entry **must** be an absolute path served by Express static middleware (e.g. `/uploads/abc123.png`), matching the existing static route already configured in `backend/src/app.js`.

### 4.4 Frontend — GameContext

15. The `SET_SCORES` action **must** store the full enriched scores array (with `submissionId`, `imageUrl`, `voterScore`, `aiScore`, `combinedScore`, `rank`) — no changes needed to the reducer itself, only the data shape changes.

### 4.5 Frontend — ResultsPage Data Source

16. The `ResultsPage` component **must** remove the `MOCK_RESULTS` constant entirely.
17. On mount, `ResultsPage` **must** first attempt to read `state.scores` from `GameContext`. If `state.scores` is populated (length > 0), it uses that data directly (the player just came from the voting phase).
18. If `state.scores` is empty (e.g. the player refreshed the page), `ResultsPage` **must** call `GET /api/challenges/:code/results` and populate the display from the API response.
19. While the API fetch is in-flight, `ResultsPage` **must** show a loading skeleton or spinner in place of the rankings content.
20. If the API returns a non-200 response, `ResultsPage` **must** show an error message: *"Results could not be loaded. The session may have expired."* with a link back to `/`.

### 4.6 Frontend — ResultsPage Display

21. Each result entry's `imageUrl` **must** be used as the `src` for podium thumbnails and the expand modal — no hardcoded `picsum.photos` URLs.
22. The **Voter Score** column in the AI Judge section **must** display `voterScore` from the scores payload.
23. The **AI Score** column **must** display `aiScore` from the scores payload.
24. The combined score shown on the podium blocks and in the rankings list **must** display `combinedScore` from the payload (not recalculated on the frontend).
25. Players with `imageUrl: null` (no submission) **must** render the existing "No Submission" placeholder in the podium and expand modal.

### 4.7 Frontend — New API Helper

26. A new function **must** be added to `frontend/src/services/api.js`:
    ```js
    export const getResults = (code) => api.get(`/challenges/${code}/results`)
    ```

## 5. Non-Goals (Out of Scope)

- Implementing a real AI judge (Claude API call) — this is a future task. The stub is sufficient.
- Persisting results to a database — in-memory `Room` storage is sufficient for the hackathon.
- Handling room expiry or cleanup — rooms persist in memory until the server restarts.
- Player authentication or session tokens.
- Showing a replay of the voting sequence on the results page.

---

## 6. Design Considerations

- The ResultsPage layout, podium, AI Judge section, and expand modal are **already built** — no visual changes are needed. Only the data source changes.
- The loading state should match the existing page background (`#f9fafb`) with a simple centered spinner or skeleton row to avoid layout shift.
- `imageUrl` values of `null` or empty strings should fall back to the existing "No Submission" placeholder already implemented in the component.

### Data Flow
```
[Voting ends]
    → server: computeScores() called inside advanceSlide()
    → server: scores stored on room.finalScores
    → server: broadcasts SHOW_RESULTS with enriched scores array
    → frontend: useSocket handler dispatches SET_SCORES
    → frontend: SET_PHASE 'results' → navigates to /results/:code
    → ResultsPage: reads state.scores (populated from socket)
    → ResultsPage: renders real data

[Player refreshes /results/:code]
    → ResultsPage: state.scores is empty
    → ResultsPage: calls GET /api/challenges/:code/results
    → server: returns room.finalScores
    → ResultsPage: renders real data
```

### AI Judge Stub — Implementation Reference
```js
// backend/src/services/aiJudge.js
// TODO: Replace mock with real Claude API call — see docs/development/prd.md §3.5
function scoreSubmission(submissionId) {
  // Deterministic hash so repeated calls return the same score
  let hash = 0
  for (const ch of (submissionId || 'none')) {
    hash = (hash * 31 + ch.charCodeAt(0)) & 0xffff
  }
  // Maps to range [3.0, 5.0] with one decimal place
  return Math.round((3.0 + (hash / 0xffff) * 2.0) * 10) / 10
}
module.exports = { scoreSubmission }
```

## 8. Success Metrics

- After completing a full game session, the ResultsPage displays the correct winner (the player whose designs received the highest average votes).
- Each top-3 podium slot shows the actual uploaded image from that player's submission (not a placeholder URL).
- Refreshing `/results/:code` in the browser re-loads the same standings without re-playing the game.
- The AI Judge section shows a non-zero, consistent score for each player (same score on refresh).
- No `MOCK_RESULTS`, `picsum.photos`, or hardcoded nicknames remain in `ResultsPage.jsx`.

---

## 9. Open Questions

- **Room expiry**: In-memory rooms persist until server restart. If the server restarts between game end and a refresh, results are lost. Should a TTL or LRU eviction be added to the store? *(Deferred — out of scope for hackathon)*
- **Tie-breaking**: If two players have the same `combinedScore`, how should rank be assigned? *(Current behavior: sort is unstable — order is insertion order. Acceptable for now.)*
- **Image CDN**: For production, `/uploads/` should be replaced with a CDN URL (S3/Cloudinary). The `imageUrl` field in the scores payload is the single place to update when this changes. *(Documented in PRD §3.A as future upgrade path)*
